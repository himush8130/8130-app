# 8130 APP — מסמך אפיון שלב א'

## 1. סקירה כללית

מערכת פנימית לניהול מערך טכנאים שנותן מענה לאנשי שטח. אנשי השטח שולחים טפסי דיווח תקלה דרך פלטפורמה חיצונית (**Base44**) שמועברים אלינו ב-Webhook, והמערכת שלנו אחראית על:

- קבלת הקריאות והקצאתן אוטומטית לטכנאי המתאים
- ניהול מחזור החיים של קריאת השירות
- ניהול קטלוג חלקי חילוף ותנועות מלאי
- ממשקי עבודה ייעודיים לטכנאי, מנהל ומחסנאי

**סקייל יעד:** ~40 רכבים, ~6,000 חלקי חילוף, ~40 טכנאים.

**עיקרון מנחה רוחבי:** **אין שדות חוסמים בשום מקום במערכת.** כל הקלט מתקבל; חריגות מסומנות ומוצגות כהתראה, אך אינן עוצרות זרימה.

**שפה:** עברית בלבד עם RTL מלא.

## 2. משתמשים והרשאות

שלוש רמות תפקיד (`role`) בתוך אפליקציה אחת, כשלכל תפקיד יש דף בית ייעודי:

| תפקיד | פעולות עיקריות |
|---|---|
| **טכנאי** (technician) | רואה את הקריאות שלו, מעדכן הערות, מבקש חלקים, סוגר קריאות |
| **מנהל** (manager) | שליטה מלאה: שינוי `profession_id` של קריאה, עריכה, ביטול, פתיחה מחדש, ניהול רכבים, צפייה בכל הקריאות והדוחות |
| **מחסנאי** (warehouse) | ניהול קטלוג חלקים, אישור בקשות הוספת חלק, תור הזמנות, רישום יציאות מהמחסן |

**אימות:**
- שלב א': מספר עובד בלבד (ללא סיסמה)
- שלב ב' (עתידי): הוספת קוד OTP ב-SMS לאחר הזנת מספר העובד

## 3. ישויות נתונים מרכזיות

### 3.1 מקצועות (`professions`)
טבלה משותפת המקשרת בין רכב לטכנאי. **סוג רכב = מקצוע טכנאי**.

| שדה | תיאור |
|---|---|
| `id` | PK |
| `name` | שם המקצוע (עברית) |

**רשימת פיתוח ראשונית:** רכב, חשמל, אופטיקה. תורחב/תצומצם בהמשך.

### 3.2 עובדים (`employees`)

| שדה | תיאור |
|---|---|
| `employee_number` | מזהה ייחודי, משמש להתחברות |
| `name` | שם העובד |
| `phone` | טלפון |
| `profession_id` | FK → professions |
| `role` | enum: technician / manager / warehouse |

**פעיל/לא פעיל לא נשמר כשדה** — נגזר דינמית מטבלת הזמינות (סעיף 3.3).

### 3.3 זמינות עובדים (`employee_availability`)
מודל דיגיטלי פשוט ליום: ברירת מחדל זמין, רישום נקודתי לימי אי-זמינות.

| שדה | תיאור |
|---|---|
| `employee_number` | FK |
| `date` | תאריך |
| `available` | boolean (ברירת מחדל true; רישומים בפועל = ימי אי-זמינות) |

### 3.4 רכבים (`vehicles`)

| שדה | תיאור |
|---|---|
| `vehicle_number` | מזהה ייחודי (מגיע מהטופס) |
| `type_id` | FK → professions (שדה שמכתיב סיווג אוטומטי של קריאות לרכב זה) |
| `model` | דגם |

CRUD על-ידי המנהל. הטבלה כמעט סטטית בפועל אך פתוחה לשינוי.

### 3.5 קריאות שירות (`service_calls`)

| שדה | תיאור |
|---|---|
| `id` (UUID) | מזהה פנימי |
| `display_id` | פורמט `SR-YY-NNNN` (למשל `SR-26-0001`), מתאפס בכל שנה |
| `external_id` | מזהה מטופס המקור (אם הגיע) |
| `vehicle_number` | כפי שהגיע בטופס (לא בהכרח קיים בטבלת רכבים) |
| `vehicle_name` | מהטופס |
| `reporter_name` | מהטופס |
| `reporter_phone` | מהטופס |
| `description` | תיאור חופשי מהטופס |
| `status` | enum (סעיף 4) |
| `profession_id` | FK → professions (קובע איזה צוות רואה את הקריאה; null בחריגה דחופה של רכב לא ידוע) |
| `created_at`, `updated_at`, `closed_at` | חותמות זמן |
| `closed_by` | מספר עובד של הסוגר |
| `anomaly_flags` | רשימת חריגות "רכות" (מידע למנהל) |

כל שדות הטופס שהגיעו מ-Base44 הם **אופציונליים**.

**הערה: אין שדה "טכנאי משויך"** — קריאה גלויה לכל הטכנאים מהמקצוע (`profession_id`), ואין "בעלים". כל אחד מהם יכול להוסיף הערות, לבקש חלקים, ולסגור. רק `closed_by` נרשם בסגירה לתיעוד.

### 3.6 חלקי חילוף (`parts`)

| שדה | תיאור |
|---|---|
| `sku` | מק"ט |
| `name` | שם |
| `quantity` | כמות במלאי |
| `location` | מיקום במחסן |
| `min_threshold` | סף מלאי נמוך |
| `supplier` | ספק |
| `pending_approval` | boolean (חלק שטכנאי הציע, ממתין לאישור מנהל/מחסנאי) |

**טעינה ראשונית לפיתוח:** 10 חלקי דמה. ייבוא 6,000 פריטים אמיתיים — שיטה תוגדר בהמשך.

### 3.7 חלקים נדרשים לקריאה (`call_required_parts`)
מה שהטכנאי הצהיר שהוא צריך. **אינו מנכה מלאי.**

| שדה | תיאור |
|---|---|
| `call_id` | FK |
| `part_sku` | FK |
| `quantity` | כמות מבוקשת |
| `status` | enum: in_stock / awaiting_order / awaiting_receipt / received |
| `requested_by`, `requested_at` | מי ביקש ומתי |

### 3.8 יציאות מהמחסן (`part_withdrawals`)
תנועת מלאי בפועל: ברגע שהטכנאי לוקח פיזית חלק מהמחסנאי. **בנקודה זו המלאי מנוכה.**

| שדה | תיאור |
|---|---|
| `call_id` | FK |
| `part_sku` | FK |
| `quantity` | כמות שיצאה |
| `withdrawn_by` | טכנאי |
| `released_by` | מחסנאי |
| `withdrawn_at` | חותמת זמן |

### 3.9 הערות לקריאה (`call_comments`)
לוג כרונולוגי, ללא דריסה.

| שדה | תיאור |
|---|---|
| `call_id` | FK |
| `author_employee_number` | כותב |
| `text` | תוכן |
| `created_at` | חותמת זמן |

תמונות — שלב ב'.

## 4. מחזור חיים של קריאת שירות

### 4.1 סטטוסים

```
חדשה → בטיפול → (ממתינה לחלקים) ↔ → סגורה
                                    ↓
                                  בוטלה
```

| סטטוס | משמעות |
|---|---|
| **חדשה** (new) | קריאה התקבלה אך הסיווג האוטומטי נכשל (חריגה דחופה לטיפול מנהל) |
| **בטיפול** (in_treatment) | מקצוע סווג; הקריאה גלויה לכל הטכנאים הזמינים מהמקצוע |
| **ממתינה לחלקים** (waiting_for_parts) | יש חלק נדרש שאינו במלאי |
| **סגורה** (closed) | הטיפול הסתיים |
| **בוטלה** (cancelled) | המנהל ביטל את הקריאה |

מאחר שהסיווג אוטומטי, קריאות מוצלחות נכנסות ישירות ל"בטיפול" ולא יוצרות מצב "חדשה" אמיתי.

### 4.2 דחיפות וסדר טיפול
**אין דחיפות. אין תור לפי סדר.** כל הקריאות מטופלות במקביל.

### 4.3 סגירת קריאה
- **מי:** כל טכנאי או מנהל. נשמר `closed_by`.
- **שדות חובה:** אין. כל השדות אופציונליים.
- **שדות אפשריים:** תיאור הטיפול (טקסט חופשי), שעות עבודה, רשימת חלקים שהתקבלו (0..N עם כמויות).
- **השפעה על מלאי:** **אין** בנקודת הסגירה — המלאי כבר נוכה ב-`part_withdrawals` כשהטכנאי לקח פיזית.
- מנהל יכול להחזיר קריאה סגורה ל"בטיפול" בכל עת.

## 5. אלגוריתם סיווג אוטומטי

ברגע קבלת טופס מ-Base44:

1. חיפוש `vehicle_number` בטבלת הרכבים.
2. אם נמצא → קריאת `type_id` → זה ה-`profession_id` של הקריאה.
3. בדיקה שיש לפחות טכנאי אחד מהמקצוע (`role='technician'`) שזמין היום (לפי `employee_availability`).
4. אם כן → סטטוס "בטיפול". הקריאה גלויה לכל הטכנאים הזמינים מהמקצוע — אין "בעלים", אין שיוך נקודתי.

**אין רוטציה. אין "מי לוקח".** כל הטכנאים הרלוונטיים רואים את הקריאה ב"קריאות שלי", ויכולים לעבוד עליה במקביל או בחלוקת עבודה הסכמית בין הצוות.

**מקרי כשל ↑ "חריגה דחופה":**
- מספר רכב לא קיים במערכת → `profession_id = NULL`, סטטוס "חדשה"
- אין טכנאי משויך למקצוע במערכת → `profession_id` מסומן, סטטוס "חדשה"
- כל הטכנאים מהמקצוע לא זמינים היום → `profession_id` מסומן, סטטוס "חדשה" (יוצג גם לטכנאים בעת שיהיו זמינים, אך מסומן למנהל לעת עתה)

חריגה דחופה ⇒ קריאה בסטטוס "חדשה", המנהל מקבל אותה ב"תור חריגות" עם פעולות אפשריות:
- **להוסיף רכב חדש** (אם מספר הרכב לא קיים)
- **לתקן מספר רכב** (טעות בהקלדה)
- **לסווג ידנית למקצוע** (אם המנהל יודע יותר מהמערכת)
- **לדחות לתאריך אחר**
- **להתעלם / לארכב**

## 6. זרימת חלקי חילוף

### 6.1 שלבי חיים של חלק נדרש בקריאה

```
טכנאי בוחן תקלה
      ↓
מוסיף פריטים ל-call_required_parts
      ↓
לכל פריט - בדיקת מלאי:
   ┌──────────┴──────────┐
יש מלאי              אין מלאי
   ↓                     ↓
in_stock         awaiting_order  ← הקריאה מקבלת סטטוס "ממתינה לחלקים"
                       ↓
                 awaiting_receipt  ← מחסנאי הזמין
                       ↓
                    received       ← הסחורה הגיעה
                       ↓
                  in_stock (לוגית)
   ↓                     ↓
       טכנאי ניגש למחסן ולוקח את החלק
                ↓
       רישום ב-part_withdrawals
                ↓
       ניכוי בפועל מ-parts.quantity
                ↓
       עבודה על הרכב, סגירת קריאה
```

### 6.2 שתי רשימות לכל קריאה
1. **חלקים נדרשים** — מה שהטכנאי ביקש (לא משפיע על מלאי)
2. **חלקים שהתקבלו** — מה שיצא בפועל מהמחסן (משפיע על מלאי)

ייתכנו פערים בין השתיים — לגיטימי. למשל: ביקש 4 ברגים, לקח 3.

### 6.3 ניהול הקטלוג
- **מנהל ומחסנאי**: CRUD מלא.
- **טכנאי**: יכול לבקש הוספת חלק חדש (`pending_approval=true`). מחסנאי או מנהל מאשרים.
- **התראת מלאי נמוך**: בשלב א' — תצוגה ויזואלית בלבד למחסנאי. ללא התראה אקטיבית.

## 7. קבלת טפסים מ-Base44 (Webhook)

### 7.1 שדות הטופס (כל השדות אופציונליים)
- שם רכב
- מספר רכב
- שם המדווח
- טלפון המדווח
- תיאור חופשי של התקלה
- מזהה ייחודי לטופס (מתוך Base44)

### 7.2 אבטחה
- שלב א': **Bearer Token בכותרת** (`Authorization: Bearer <secret>`).
- שלבים מתקדמים יותר (HMAC, IP whitelist) — בעתיד אם יידרש.

### 7.3 Idempotency
לא בשלב א'. אם תהיה בעיה אמיתית של כפילויות — נטפל אז (וכנראה ע"י external_id).

### 7.4 חריגות בקבלה
**עיקרון: השרת תמיד מקבל. לעולם לא דוחה.**

| סוג חריגה | התנהגות |
|---|---|
| **רכה** (למשל מספר טלפון חסר, מדווח חסר) | קריאה משויכת ומטופלת רגיל. הערה למנהל "בקריאה X חסר Y" — מידע, לא חוסם. |
| **דחופה** (סיווג אוטומטי נכשל) | קריאה בסטטוס "חדשה" עם דגל חריגה. מופיעה בתור חריגות אצל המנהל. |

### 7.5 סימולטור Base44
- תיקיית `simulator/templates/` עם קבצי JSON מוכנים מראש.
- סקריפט `simulator/send.py` (או `.js`) שמקבל פרמטר:
  - `--template <name>` — שולח טופס ספציפי
  - `--random` — בוחר תבנית רנדומלית
- שולח POST ל-Webhook עם ה-Bearer Token.
- מאפשר פיתוח עצמאי של הצד שלנו לפני ש-Base44 פותחים את הצד שלהם.

## 8. ממשקים ומסכים (Highlights)

### 8.1 דף בית טכנאי
- רשימת הקריאות שלו עם סינון לפי סטטוס/רכב/תאריך
- כניסה לכרטיס קריאה: עדכון הערות, ניהול חלקים נדרשים, סגירה
- **גישה offline (קריאה בלבד)**: יראה את הקריאות שלו גם בלי אינטרנט; עריכות דורשות אינטרנט

### 8.2 דף בית מנהל
- רשימת קריאות פתוחות
- תור חריגות דחופות
- עומס נוכחי לפי טכנאי/מקצוע
- ניהול רכבים, ניהול עובדים, ניהול זמינות
- שליטה מלאה: שינוי profession_id, עריכה, ביטול, פתיחה מחדש

### 8.3 דף בית מחסנאי
- חלקים בסטטוס מלאי נמוך (תצוגה ויזואלית)
- תור חלקים שמחכים להזמנה / להגעה
- ניהול קטלוג: CRUD, אישור בקשות הוספה
- רישום יציאות מהמחסן (`part_withdrawals`)

### 8.4 כרטיס רכב
- פרטי רכב + היסטוריית כל הקריאות עליו לאורך כל הזמן (נשמר טבעית — קריאות לא נמחקות).

### 8.5 חיפוש וסינון בקריאות
סינון לפי שדות: סטטוס, מקצוע, רכב, טכנאי, טווח תאריכים. ללא חיפוש טקסט חופשי בשלב א'.

### 8.6 מחסן יחיד
שלב א' — מחסן מרכזי אחד בלבד. ריבוי מחסנים / מחסן רכבי שירות — בעתיד.

### 8.7 התראות לטכנאי
- שלב א': רענון ידני בעת כניסה לאפליקציה.
- שלב ב': Push Notifications של PWA (opt-in).
