1. סקירה כללית

אנשי שטח שולחים טפסי דיווח תקלה דרך פלטפורמת Base44; הטופס מגיע ל-Webhook אצלנו, והמערכת אחראית על: קבלת הקריאות וסיווגן האוטומטי, ניהול מחזור החיים שלהן, ניהול קטלוג חלקי חילוף ותנועות מלאי, וממשקי עבודה לשלושה תפקידים.

עיקרון מנחה רוחבי: אין שדות חוסמים בשום מקום במערכת. כל הקלט מתקבל; חריגות מסומנות ומוצגות כהתראה אך אינן עוצרות זרימה.

אימות שלב א': מספר עובד בלבד. שלב ב' — OTP ב-SMS.

5. אלגוריתם הסיווג האוטומטי

רץ ברגע קבלת טופס מ-Base44. שלוש החלטות, שלושה סוגי כשל ↑ "חריגה דחופה".

תור החריגות אצל המנהל: הוסף רכב חדש · תקן מספר רכב · סווג ידנית · דחה לתאריך · התעלם / ארכב.

אין רוטציה. אין "מי לוקח". הקריאה גלויה לכל הטכנאים מהמקצוע, והם מסתדרים ביניהם.

6. זרימת חלקי חילוף

פייפליין: בקשה → המתנה → קבלה → יציאה פיזית. רק השלב האחרון מנכה מלאי.

שתי רשימות לכל קריאה

  • חלקים נדרשים — מה שהטכנאי ביקש (לא משפיע על מלאי)
  • חלקים שהתקבלו — מה שיצא בפועל מהמחסן (משפיע)

פערים בין השתיים — לגיטימי. ביקש 4 ברגים, לקח 3.

סף מלאי נמוך

שלב א' — תצוגה ויזואלית בלבד למחסנאי.
שלב ב' — התראה אקטיבית.

7. קבלת טפסים מ-Base44

השרת תמיד מקבל. לעולם לא דוחה.

סוג חריגההתנהגותסטטוס סופי
רכה למשל מספר טלפון חסר, מדווח חסר. הקריאה משויכת ומטופלת רגיל. הערה למנהל "בקריאה X חסר Y". בטיפול
דחופה סיווג אוטומטי נכשל (3 מקרים). דגל חריגה. מופיעה בתור חריגות אצל המנהל. חדשה
אבטחה שלב א': Bearer Token ב-Authorization. HMAC / IP whitelist — בעתיד אם יידרש.
Idempotency: לא בשלב א'. אם תהיה בעיה אמיתית של כפילויות — יטופל אז (כנראה לפי external_id).

סימולטור Base44

  • simulator/templates/ — קבצי JSON מוכנים מראש
  • simulator/send.py --template <name> — שולח טופס ספציפי
  • simulator/send.py --random — בוחר תבנית רנדומלית
